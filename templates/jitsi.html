Cloudflare Zero Trust + Jitsi Meet Setup
{% if settings.cf_mode == 'tcp_only' %} {# ── TCP-only: manual step + automation summary ── #}

You have selected TCP-only mode. All traffic goes through the Cloudflare tunnel. Media falls back to TURN over TCP (5349/tcp). No firewall changes needed.

1. Create Cloudflare Tunnel Route Manual

In the Cloudflare Zero Trust dashboard, create a public hostname in your tunnel:

Hostname: {{ settings.hostname or 'meet.example.com' }}
Service:  https://{{ '<guest-internal-ip>' }}:443
TLS:      No TLS Verify (self-signed cert on guest)

# Under Additional Settings:
HTTP Host Header: {{ settings.hostname or 'meet.example.com' }}
2. Click "Configure for Cloudflare" Automated

The configure button will automatically:

  • Enable TCP transport in jvb.conf
  • Disable P2P and force WebSocket bridge channel in {{ settings.hostname or 'meet.example.com' }}-config.js
  • Verify Prosody advertises TURN (turns entry in external_services)
  • Verify coturn is listening on 5349/tcp
  • Restart all Jitsi services

Select a Cloudflare Zero Trust mode in the Configuration panel and save to enable automated configuration.

Cloudflare tunnels only carry TCP/HTTP traffic. Jitsi's video bridge (JVB) uses 10000/udp for media, which cannot traverse a tunnel. Two options:

Option A: Tunnel everything (TCP-only, easiest)
All traffic goes through Cloudflare. Media falls back to TURN over TCP. Slightly higher latency but no firewall changes needed.

Option B: Tunnel HTTP + direct UDP (best quality)
Web UI through Cloudflare tunnel, media direct over UDP. Requires opening 10000/udp on your firewall/NAT to the Jitsi guest.

1. Cloudflare Tunnel Route

Create a public hostname in your Cloudflare tunnel pointing to the Jitsi guest:

Hostname: {{ settings.hostname or 'meet.example.com' }}
Service:  https://{{ '<guest-internal-ip>' }}:443
TLS:      No TLS Verify (self-signed cert on guest)

# Under Additional Settings:
HTTP Host Header: {{ settings.hostname or 'meet.example.com' }}
2. JVB NAT Harvester (Option B only)

Tell JVB its public IP so it can send correct ICE candidates. Edit on the Jitsi guest:

# /etc/jitsi/videobridge/sip-communicator.properties
org.ice4j.ice.harvest.NAT_HARVESTER_LOCAL_ADDRESS=<guest-internal-ip>
org.ice4j.ice.harvest.NAT_HARVESTER_PUBLIC_ADDRESS=<your-public-ip>

# Then restart:
sudo systemctl restart jitsi-videobridge2

Skip this step for Option A (tunnel-only) — all media goes through TURN.

3. Configure coturn (TURN relay)

coturn is installed by jitsi-meet-turnserver. Verify/edit its config:

# /etc/turnserver.conf
listening-port=3478
tls-listening-port=5349
fingerprint
lt-cred-mech
use-auth-secret
static-auth-secret=<same-as-prosody-turncredentials_secret>
realm={{ settings.hostname or 'meet.example.com' }}
cert=/etc/jitsi/meet/{{ settings.hostname or 'meet.example.com' }}.crt
pkey=/etc/jitsi/meet/{{ settings.hostname or 'meet.example.com' }}.key
no-multicast-peers

# Allow the Jitsi guest's internal subnet for relay:
allowed-peer-ip=10.0.0.0/8

# Then restart:
sudo systemctl restart coturn
4. Advertise TURN to clients via Prosody

Prosody tells clients where the TURN server is. Verify the external_services block:

# /etc/prosody/conf.d/{{ settings.hostname or 'meet.example.com' }}.cfg.lua
-- Inside the VirtualHost section, look for or add:
external_services = {
    { type = "stun", host = "{{ settings.hostname or 'meet.example.com' }}", port = 3478 },
    { type = "turn", host = "{{ settings.hostname or 'meet.example.com' }}", port = 3478, transport = "udp", secret = true, ttl = 86400, algorithm = "turn" },
    { type = "turns", host = "{{ settings.hostname or 'meet.example.com' }}", port = 5349, transport = "tcp", secret = true, ttl = 86400, algorithm = "turn" },
};

-- The secret is read from:
-- /etc/jitsi/meet/{{ settings.hostname or 'meet.example.com' }}-oauthl-oauthSecret.key
-- (auto-generated during install, shared with coturn)

# Then restart:
sudo systemctl restart prosody
5. Nginx WebSocket proxy

Ensure the Jitsi nginx config proxies WebSocket paths (usually auto-configured):

# /etc/nginx/sites-available/{{ settings.hostname or 'meet.example.com' }}.conf
# These blocks should already exist after install:
location /xmpp-websocket {
    proxy_pass http://127.0.0.1:5280/xmpp-websocket;
    proxy_http_version 1.1;
    proxy_set_header Upgrade $http_upgrade;
    proxy_set_header Connection "upgrade";
}

location /colibri-ws/ {
    proxy_pass http://127.0.0.1:9090/colibri-ws/;
    proxy_http_version 1.1;
    proxy_set_header Upgrade $http_upgrade;
    proxy_set_header Connection "upgrade";
}

7. Force TCP-only (Option A)

Force all media through TURN/TCP (no direct UDP). Disable P2P and enable TCP transport in JVB:

# /etc/jitsi/videobridge/jvb.conf
# Add inside the videobridge { } block:
ice {
    udp {
        port = 10000
    }
    tcp {
        enabled = true
        port = 4443
    }
}

# /etc/jitsi/meet/{{ settings.hostname or 'meet.example.com' }}-config.js
# Force TURN by setting:
config.p2p = { enabled: false };
config.openBridgeChannel = 'websocket';

After any config changes, restart all Jitsi services: sudo systemctl restart prosody jicofo jitsi-videobridge2 coturn nginx
